# 📚 Fuck-Leetcode

📓 个人 LeetCode 刷题笔记仓库：按题型整理的 Markdown，包含题目链接、思路与参考代码（以 Python 为主）。

## 🎯 为什么要做这份笔记

对不少计算机专业的同学来说，学校常规课程里很少系统接触算法题；到了找实习、找工作时，又往往不得不短期内开始大量刷题。作者自己也曾经断断续续刷过一些题，但缺少完整的知识体系，很多题过一段时间就忘了该怎么做。

另一方面，对大多数普通人而言，我们未必有「一眼看穿本质」的天赋，也没有 OI 选手那样长期训练出来的直觉。不少题目看完题解会觉得解法很精妙，可换自己独立做时，就是想不到该往哪个方向想。

因此本仓库里的笔记，更侧重两件事：

1. 💡 **思路部分尽量短**，不写成长篇标准题解；重点是尝试说明：**如果从零开始、只凭题面信息，可以怎样一步步联想到最终解法**，而不是直接呈现「标准答案式」的推理链。  
   把思路写短还有一层现实考虑：**复习旧题**时，人们通常不会乐意对着大段文字从头啃（尽管那样写往往更标准、更完整），反而更希望几眼里抓住**可复用的核心钩子**；所以正文会有意压篇幅，优先保留「下次见到类似题面该往哪想」那一点，而不是把笔记写成第二份官方题解。
2. 📖 **「焚诀」等章节开头的总结**，不只做「题面 → 算法类型」这一件事：  
   - 🔎 **识别套路**：从**题目的外在特征**反推该考虑哪类算法。教科书式的定义（例如动态规划用于「子问题重叠 + 子问题间存在最优选择」）在纸面上都对，但对着一道具体题时，很多人并不会立刻往那么抽象的方向去想。相比之下，若先建立更「表层」的联想——例如看到子序列、子数组、子字符串相关的最值问题就优先想到动态规划——往往更容易在实战中把题对上号。  
   - 🧱 **沉淀写法与规律**：把一类题里**反复出现的代码写法、边界条件和题型规律**单独拎出来归纳，避免每道题都在实现细节上从零试错。例如动态规划中**背包问题**相关部分，对 **DP 数组（含维度与下标含义）应如何初始化**做了较系统的整理。

如果你认同这种「从特征到套路、再从套路到具体状态与转移」的路径，这份笔记可能会比官网上许多偏「展示结论」的题解更顺手一些。

## 🛠️ 小工具

在项目根目录执行：

**查询某道题是否已收录**

```bash
python check.py -d <题号> -t <题目标题>
```

**统计或浏览已整理题目**

```bash
python count.py
```
